## Requirements

### Requirement: Codebase analysis detects capabilities

The `specflow.spec` command SHALL analyze the project codebase including file structure, configuration files, and source code internals (functions, classes, API endpoints) to detect capability candidates.

#### Scenario: Detect capabilities from directory structure and code

- **WHEN** the user runs `/specflow.spec`
- **THEN** the system SHALL scan the project directory structure, read configuration files (package.json, go.mod, Cargo.toml, pyproject.toml, etc.), and read key source files (entry points, routers, model definitions) to produce a list of capability candidates

#### Scenario: Empty or minimal project

- **WHEN** the user runs `/specflow.spec` on a project with no recognizable structure
- **THEN** the system SHALL report that no capabilities were auto-detected and prompt the user to manually define capabilities

### Requirement: User selects capabilities for spec generation

The system SHALL present the detected capability list to the user and allow selection, addition, and removal before generating specs.

#### Scenario: User selects from detected capabilities

- **WHEN** the system presents the detected capability list
- **THEN** the user SHALL be able to select which capabilities to generate specs for using AskUserQuestion with multiSelect

#### Scenario: User adds a capability not detected

- **WHEN** the user selects "Other" during capability selection
- **THEN** the system SHALL accept a custom capability name and include it in the generation list

#### Scenario: User removes a detected capability

- **WHEN** the user deselects a capability from the presented list
- **THEN** the system SHALL exclude that capability from spec generation

### Requirement: Interactive spec generation per capability

The system SHALL ask clarifying questions for each selected capability and generate `openspec/specs/<name>/spec.md` with the answers integrated.

#### Scenario: Questions asked per capability

- **WHEN** spec generation starts for a selected capability
- **THEN** the system SHALL ask questions about scope, key requirements (WHEN/THEN scenarios), and constraints/preconditions using AskUserQuestion one at a time

#### Scenario: Spec file created with correct format

- **WHEN** all questions for a capability are answered
- **THEN** the system SHALL create `openspec/specs/<name>/spec.md` containing a Purpose section, Requirements with `### Requirement:` headers, and `#### Scenario:` blocks in WHEN/THEN format

### Requirement: OpenSpec CLI integration with fallback

The system SHALL prioritize OpenSpec CLI features for spec creation and fall back to agent-driven generation when CLI features are unavailable.

#### Scenario: CLI spec support available

- **WHEN** `openspec new spec` and `openspec instructions` commands support spec artifact type
- **THEN** the system SHALL use CLI-provided templates and instructions to generate the spec

#### Scenario: CLI spec support unavailable

- **WHEN** `openspec new spec` or `openspec instructions` does not support spec artifacts
- **THEN** the system SHALL use the format of existing specs in `openspec/specs/*/spec.md` as a reference template and generate the spec content through agent analysis

### Requirement: Specflow main flow handoff on missing specs

The `/specflow` command SHALL check for existing specs before proposal creation and offer to redirect to `specflow.spec` when none exist.

#### Scenario: No specs exist when running specflow

- **WHEN** the user runs `/specflow` and `openspec/specs/` contains no spec directories
- **THEN** the system SHALL display a prompt via AskUserQuestion offering to run `/specflow.spec` first or continue without specs

#### Scenario: Specs already exist

- **WHEN** the user runs `/specflow` and `openspec/specs/` contains one or more spec directories
- **THEN** the system SHALL proceed with the normal specflow workflow without interruption

#### Scenario: User skips spec generation

- **WHEN** the system offers to redirect to `/specflow.spec` and the user selects "スキップ"
- **THEN** the system SHALL continue with the normal specflow workflow

### Requirement: Generated specs enable normal specflow workflow

After spec generation, the normal specflow workflow (proposal → design → apply) SHALL function without spec delta errors.

#### Scenario: Specflow design after spec generation

- **WHEN** specs have been generated by `/specflow.spec` and the user runs `/specflow` followed by `/specflow.design`
- **THEN** the system SHALL successfully create delta specs referencing the generated baseline specs without errors

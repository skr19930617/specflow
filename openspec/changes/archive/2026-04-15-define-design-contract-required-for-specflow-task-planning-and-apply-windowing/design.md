## Context

Currently, `design.md` is generated from OpenSpec artifact instructions as a free-form document with sections: Context, Goals/Non-Goals, Decisions, Risks/Trade-offs. The task planner's `generateTaskGraph(designContent, changeId, specNames)` uses LLM inference to extract bundle boundaries and dependencies from this free-form text, which produces unstable results when the design lacks explicit concern/state/contract/boundary information.

The design generation flow works as follows:
1. `specflow-design-artifacts next` orchestrator returns the `design` artifact with its template and instruction
2. The main agent (Claude) generates `design.md` using those instructions
3. `specflow-review-design` reviews the design using prompt contracts (`review_design_prompt.md`, `review_design_rereview_prompt.md`)
4. Design prompt text lives in markdown source files (`assets/global/prompts/review_design_prompt.md`, `review_design_rereview_prompt.md`). `src/contracts/prompts.ts` defines prompt metadata (id, filePath, sourcePath, outputExample) but not prompt text. The build pipeline renders the markdown sources into `dist/package/global/prompts/`.

The review design agent (`specflow-review-design.ts`) reads artifacts via `readDesignArtifactsFromStore()`, sends them to the review agent, and persists findings in a review ledger. The existing review flow handles `request_changes` → fix → re-review cycles.

## Goals / Non-Goals

**Goals:**
- Define 7 mandatory planning-oriented section headings for `design.md`
- Add structural validation that checks heading existence and non-empty content
- Update the design generation prompt contract to request planning sections
- Add "task-plannable" quality gate to design review as high-severity findings
- Apply requirements only to changes where `design.md` does not yet exist

**Non-Goals:**
- Replacing the current design format entirely — existing free-form sections remain alongside new planning sections
- Forcing a single rigid markdown schema — section content is validated only for existence/non-emptiness, not structure
- Implementing task planner runtime changes — the task planner consuming planning sections is a separate change
- Validating content quality in the structural gate — LLM-based content review remains the review agent's responsibility

## Concerns

1. **Design planning section contract** — define the 7 mandatory headings and their minimum expected content, including N/A handling for non-applicable sections
2. **Design generation prompt update** — update the prompt contract used by design generation to instruct the agent to produce planning sections
3. **Structural validation gate** — implement heading-level validation (exists + non-empty) that runs before or as part of design review
4. **Design review integration** — surface structural validation failures as `high` severity findings with `task-plannable` category in the existing review flow
5. **Backward compatibility** — ensure existing designs are not retroactively validated against the new contract

## State / Lifecycle

- **Canonical state**: The 7 mandatory section headings are a schema property of `design.md` — they exist as markdown headings in the file. No runtime state or database entries are involved.
- **Derived state**: Structural validation result (pass/fail with missing/empty section names) is derived by parsing `design.md` headings at validation time. It is not persisted.
- **Lifecycle boundary**: The planning section contract applies from `design_draft` phase onward. It is enforced at `design_review` via the existing review flow.
- **Persistence-sensitive state**: The review ledger (`review-ledger-design.json`) stores `task-plannable` category findings alongside existing review findings. No separate persistence is needed. Note: `task-plannable` is outside the LLM review agent's output category enum (`completeness | feasibility | ordering | granularity | scope | consistency | risk`). This is intentional — `task-plannable` findings are injected programmatically by the structural validator, not produced by the LLM. The existing `ReviewFinding` type's `category` field is typed as `string`, which accommodates this without schema changes.

## Contracts / Interfaces

### Inputs consumed
- `design.md` — the artifact being validated for planning section presence
- `review_design_prompt.md` / `review_design_rereview_prompt.md` — prompt contracts that instruct the review agent what to check
- OpenSpec artifact instruction for `design` — the generation-time instruction that tells the main agent what sections to produce

### Outputs produced
- Updated `design.md` with 7 planning section headings (generated by design agent)
- `task-plannable` category findings in `review-ledger-design.json` (produced by review flow)
- Updated prompt markdown sources in `assets/global/prompts/review_design_prompt.md` and `review_design_rereview_prompt.md` (build-time artifacts)
- Updated prompt metadata in `src/contracts/prompts.ts` if schema changes are needed (e.g., adding `task-plannable` to output example category union)

### Cross-bundle interfaces
- The structural validator is a pure function: `validatePlanningHeadings(designContent: string) → { valid: boolean, missing: string[], empty: string[] }`
- Review integration calls this validator and converts failures to `ReviewFinding[]` objects before ledger persistence
- The design generation prompt contract references the 7 heading names as an enumerated list

## Persistence / Ownership

- **Planning section heading list**: Owned by `design-planning-contract` spec. Stored as a constant array in source code (e.g., `src/lib/design-planning-headings.ts`). This is the single source of truth for heading names used by both generation and validation.
- **Structural validation logic**: Owned by `design-planning-contract` spec. Implemented as a pure function in a new module.
- **Review prompt updates**: Owned by `contract-driven-distribution` spec. The prompt text in `assets/global/prompts/review_design_prompt.md` and `review_design_rereview_prompt.md` references the heading list. `src/contracts/prompts.ts` owns only prompt metadata (id, paths, output schema example).
- **Review finding integration**: Owned by `review-orchestration` spec. The review pipeline in `specflow-review-design.ts` calls the structural validator and adds findings.
- **Backward compatibility check**: Owned by `review-orchestration` spec. Determined by the review round number from the review ledger — the task-plannable gate runs only on round 1 (first review after generation). Pre-existing designs that have already passed review will not encounter a round 1 review for this gate.

## Integration Points

- **OpenSpec artifact instruction system**: The design artifact `instruction` field (returned by `specflow-design-artifacts next`) needs to include planning section requirements. The design generation instructions are defined in `src/contracts/command-bodies.ts` (the `/specflow.design` command body that orchestrates artifact generation). `openspec/config.yaml` contains only `schema` and `diff_warn_threshold` — it has no `rules` section.
- **Prompt rendering pipeline**: Prompt text lives in `assets/global/prompts/review_design_prompt.md` and `review_design_rereview_prompt.md`. `src/contracts/prompts.ts` defines prompt metadata (id, paths, output schema example) but not the prompt text itself. The `review_design_prompt.md` needs to instruct the review agent to check for planning sections. The build pipeline (`src/build.ts`) renders the markdown sources into `dist/package/global/prompts/`.
- **Review agent invocation**: `callReviewAgent()` in `specflow-review-design.ts` already passes design content to the review agent. The structural validation can run before or alongside the LLM-based review.
- **No external system dependencies**: All changes are within the specflow source tree.

## Decisions

### Decision 1: Structural validation as a pure function called within the review pipeline

The structural validator runs inside `specflow-review-design.ts` as a pre-check before invoking the review agent. Failures are converted to `ReviewFinding[]` objects and merged into the ledger alongside LLM-generated findings.

**Alternative considered**: Separate pre-review CLI step (e.g., `specflow-validate-design-structure`). Rejected because it would add a new phase transition and complicate the workflow state machine. The existing review flow already handles finding-based remediation.

### Decision 2: Heading names stored as a single-source constant

The 7 heading names are defined once in `src/lib/design-planning-headings.ts` and imported by both the generation prompt and the validation function. This prevents drift between what generation produces and what validation expects.

**Alternative considered**: Hardcode heading names in each location. Rejected because heading renames would require synchronized changes in multiple files.

### Decision 3: Backward compatibility via review round number

The task-plannable gate is applied only when the current review is round 1 (the first review after design generation). If the review round is > 1, the design was generated under the new instructions and has already been through the gate — no re-check is needed. For designs generated before this change was deployed, round 1 will never occur again (they have already passed review), so the gate naturally does not apply to them.

Concretely: the review ledger already tracks the review round number. The structural validator is invoked only when `reviewRound === 1`. This avoids the problem that `design.md` always exists at review time (since the review runs after generation), and requires no new metadata or ChangeArtifactStore changes.

**Alternative considered**: Check `design.md` existence or timestamps via ChangeArtifactStore. Rejected because design.md always exists at review time (it was just generated), and the store has no "existed before generation" metadata. Checking git status (untracked = new) was also considered but rejected as fragile and environment-dependent.

## Risks / Trade-offs

- **Risk: Heading name drift between prompt and validator** → Mitigation: Single-source constant imported by both.
- **Risk: Overly strict validation rejects valid designs with slightly different heading names** → Mitigation: Validation uses case-insensitive heading matching and allows heading text to contain additional words (e.g., "## Concerns and Vertical Slices" matches "Concerns").
- **Risk: N/A sections reduce design quality** → Mitigation: Content quality remains the review agent's responsibility. The structural gate only ensures headings exist with non-empty content.
- **Risk: Review prompt changes affect existing review behavior** → Mitigation: Planning section instructions are additive. Existing review criteria are preserved.
